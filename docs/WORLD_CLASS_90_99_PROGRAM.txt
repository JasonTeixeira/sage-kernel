PRINCIPLE (non-negotiable, inherited from the blueprint):
  "Nothing stated, everything proven." A category only earns its score when a
  real command/test/evidence backs it, recorded in the tamper-evident proof
  ledger. No fake green. No scaffolding. No debt code. No abandoned tasks.
  Every score in this program is EARNED or honestly marked blocked_*.

--------------------------------------------------------------------------------
0. HONEST ATTAINABILITY VERDICT (read this first)
--------------------------------------------------------------------------------
Can all 20 categories genuinely reach 90–99, earned? Mostly YES. Three have
honest ceilings we name up front and engineer around rather than fake:

  * Intelligence / Autonomy (cat 14): the HARNESS can reach 95 (diagnosis,
    perspective-diverse verification, learning, bounded retries). But autonomy
    OUTCOME quality is bounded by the underlying model. We score the HARNESS
    (provable) and report outcome quality separately and honestly. Target 95
    harness; outcomes labeled, not inflated.

  * Language coverage (cat 12): "every language" is unbounded and dishonest to
    claim. We define the target set = the languages you actually ship
    (TypeScript, JavaScript, Python, Swift). 95 = world-class on the target set,
    with honest "unsupported" for the rest. We never claim languages we cannot
    parse.

  * Performance/scalability of the tool (cat 17): 95 requires real benchmarks at
    monorepo scale with statistical gating. Attainable, but the number is only
    earned with a saved baseline + regression proof, not a vibe.

Everything else (the other 17) is attainable to 90–99 with engineering + proof.
Composite target: architecture 78 -> 94. We do NOT declare world-class until the
weighted composite is >= 92 AND no single category is < 90, each EARNED.

1. THE COMPLETION LOOP (the meta-engineering that guarantees the rest)
--------------------------------------------------------------------------------
This is the loop the system runs ON ITSELF (and on your apps). It is designed so
tasks ALWAYS reach completion, never leave debt, never abandon work.

  CONTRACT  -> every task starts as a contract: goal + acceptance criteria +
               required gates (derived from profile + risk). A task with no
               acceptance criteria is REJECTED (blocked_missing_criteria), not
               started — so we never "kind of" do work.

  PLAN      -> select the engineering loop (refactor/bugfix/feature/migration/
               incident/perf/hardening/greenfield/dep-upgrade) and resolve its
               gate set. Risk-aware: high-risk injects security+release gates.

  EXECUTE   -> run each gate. Real commands only.

  PROVE     -> every gate result -> proof ledger (hash-chained) + proof graph
               (claim -> evidence). Unbacked claim => hallucination gate fails.

  DIAGNOSE  -> on any gate failure: classify (category, location, instruction).

  SELF-HEAL -> dispatch the routed agent to fix; adversarially verify (strict
               majority, perspective-diverse); re-run the gate. Bounded retries
               with a typed StopReason — never an infinite or silent loop.

  COMPLETE  -> the loop exits ONLY with a typed StopReason:
                 completed | required_gate_failed | blocked_needs_approval |
                 needs_work | blocked_missing_criteria
               "needs_work" and "required_gate_failed" are NOT completion — they
               re-enter the loop (or escalate) until completed or an explicit,
               recorded human approval/stop. No task silently ends.

  NO-DEBT INVARIANTS (enforced as gates, every iteration):
    - zero-dead: no orphan files / unused deps (refactor/dead-code gate)
    - zero-scaffold: no TODO/stub/placeholder markers in shipped code (scan gate)
    - status-honesty: no "done/passed" without backing evidence
    - coverage floor: new code must be covered or release:check fails
    - complexity budget: no function over budget (quality:complexity gate)
    - drift: no file outside allowed scope; contracts match implementation

  WHY THIS GUARANTEES COMPLETION: the only way "out" is a typed terminal state,
  and the non-terminal states feed back in. Debt/scaffold/abandonment are each a
  FAILING GATE, so the loop cannot report success while they exist. This is the
  architecture for "tasks actually get done and code is actually clean."

4. VERIFICATION PROTOCOL (how every score is EARNED, not claimed)
--------------------------------------------------------------------------------
  - Each category has a machine-checkable PROVE criterion above. The score is
    computed from evidence, not assigned.
  - Every phase ends with: full suite green, test:coverage green, release:check
    green (now includes coverage + complexity), proof:ledger valid, and a
    recorded scorecard delta.
  - New engines ship with: unit tests + property/fuzz tests + a real run on a
    foreign repo (e.g., giggl) captured as evidence.
  - No category may rise without a new PROVE artifact in the ledger. A score
    with no positive evidence is capped (existing scoreboard rule).

--------------------------------------------------------------------------------
5. THE SELF-IMPROVEMENT ORCHESTRATION (the system builds itself, in its own loop)
--------------------------------------------------------------------------------
  Each phase is itself run through the COMPLETION LOOP (section 1):
    contract(phase goal + PROVE criteria) -> plan(gates) -> build -> prove ->
    diagnose failures -> self-heal -> complete(typed stop).
  The kernel dogfoods on itself: kernel.loop.full_cycle + review + sast + score
  gate every change; release:check is the required-before-exit gate. This is the
  recursive proof: the tool that proves software proves its own evolution.

--------------------------------------------------------------------------------
6. HONEST CEILINGS & HOW WE HANDLE THEM (so we never fake a number)
--------------------------------------------------------------------------------
  - Autonomy outcomes: scored as HARNESS quality (provable) + a separate, honest
    outcome metric (model-bound). We never report model luck as kernel quality.
  - Language coverage: scored against the declared target set; everything else is
    honestly "unsupported", never silently skipped-as-pass.
  - Perf at scale: only earned with a saved baseline + regression proof.
  - Team/CI: earned by a validated, dry-run-proven Action — not by intent.
  If any category cannot be honestly earned to 90, the program SAYS SO and caps
  it, with the reason recorded. World-class means honest, not maximal.
