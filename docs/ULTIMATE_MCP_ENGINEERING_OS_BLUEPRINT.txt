SAGE KERNEL ULTIMATE MCP ENGINEERING OS BLUEPRINT
=================================================

Purpose
-------

Build Sage Kernel into a proof-first MCP-native engineering operating system for
daily software development. The system must minimize AI hallucination, eliminate
dead scaffolding, force evidence-backed claims, audit its own work, and loop
projects through planning, implementation, testing, review, security, release,
and postmortem until outcomes are proven.

This is not a marketing roadmap. This is the build blueprint from the current
repo posture to an institutional-grade engineering OS.


Current Honest Posture
----------------------

Verified strengths:

- Local CLI works and packed temp-prefix global install passes.
- MCP server exposes 91 tools and passes manifest/smoke validation.
- SQLite/postgres adapters, queue, worker, dashboard, review, security,
  workflows, profiles, memory, drift, stress, and benchmark systems have tests.
- Full suite passes locally:
  - npm test -- --test-concurrency=1
  - 239 tests, 238 passed, 1 skipped
- zero-dead, status-honesty, qa, red-team, benchmark, stress, retrieval,
  orchestration, observability, and temp global install proof gates pass.

Verified blockers:

- Public npm package is not published:
  - npm view sage-kernel version --json returns E404.
- release:check fails because mcp:clients:prove fails.
- Claude Desktop and Cursor SDK-compatible MCP calls pass, but real UI tool-call
  proof is blocked because real app bundles/tool-call evidence are missing.
- Old template-scaffold.mjs still contains placeholder-based output and should
  be deleted or replaced.
- template-scaffold-v2.mjs still emits an AI route placeholder.
- Model-rubric evals are blocked unless SAGE_MODEL_RUBRIC_COMMAND is configured.
- Release-scale stress is not fully proven:
  - dashboard 50k / concurrency 250
  - queue 5M
  - release soak 50 cycles
- Real repo matrix exists but is weak:
  - average score 80
  - 10 ambiguous
  - 11 low-confidence


North Star
----------

Sage Kernel should become the operating layer that forces any AI or human
engineering workflow through this invariant:

  No claim is accepted unless the proof graph connects the goal to the evidence.

The system must prove:

1. The right thing was requested.
2. The requirement was converted to executable acceptance criteria.
3. The right files changed.
4. The right tests covered those files.
5. The right risk gates ran.
6. The right review/security/red-team checks ran.
7. The result was installed or exercised through the real runtime surface.
8. The score is capped honestly when external proof is missing.
9. The final answer cites evidence and refuses unsupported claims.

Layer 2: Proof Core

1. Proof Ledger

   Append-only evidence records for every command/tool/run.

   Required fields:

   - proofId
   - runId
   - parentProofIds
   - command or tool name
   - normalized input
   - inputHash
   - stdoutPath
   - stderrPath
   - artifactPaths
   - outputHash
   - gitCommit
   - gitDiffHash
   - workspaceDirtyState
   - startedAt
   - finishedAt
   - durationMs
   - exitCode
   - status
   - verifier
   - freshnessExpiresAt
   - riskLevel
   - approvalId when required

   Deliverables:

   - packages/proof/ledger.mjs
   - packages/proof/schema.json
   - scripts/proof-ledger-validate.mjs
   - MCP tools:
     - kernel.proof.record
     - kernel.proof.get
     - kernel.proof.list
     - kernel.proof.verify

   Tests:

   - Records proof for successful command.
   - Records proof for failed command.
   - Rejects malformed proof.
   - Detects tampered proof hash.
   - Supports parent/child proof relationships.
   - Preserves stdout/stderr artifacts.


2. Proof Graph

   Connect goals, requirements, files, tests, commands, tool calls, evidence,
   scores, and final claims.

   Required node types:

   - goal
   - requirement
   - acceptance_criterion
   - risk
   - changed_file
   - test
   - command
   - mcp_tool_call
   - review_finding
   - security_finding
   - redteam_case
   - release_gate
   - proof
   - claim
   - score

   Required edge types:

   - satisfies
   - tests
   - covers
   - depends_on
   - produced_by
   - verified_by
   - blocked_by
   - supersedes
   - regressed_from

   Deliverables:

   - packages/proof/graph.mjs
   - packages/proof/graph-schema.json
   - scripts/proof-graph-build.mjs
   - scripts/proof-graph-validate.mjs
   - Dashboard proof graph view.
   - MCP tools:
     - kernel.proof_graph.build
     - kernel.proof_graph.query
     - kernel.proof_graph.validate

   Tests:

   - Goal links to requirements.
   - Requirement links to acceptance criteria.
   - Changed file links to mapped tests.
   - Final claim is rejected when no proof path exists.
   - Release gate fails when required graph edges are missing.


3. Claim Firewall

   Parse generated reports, MCP outputs, release summaries, and final answers.
   Downgrade or reject unsupported claims.

   Claims requiring proof:

   - done
   - complete
   - passed
   - production ready
   - tested
   - verified
   - secure
   - released
   - installed
   - connected
   - no scaffold
   - no placeholder
   - no dead code

   Status policy:

   - passed requires proofIds and fresh evidence.
   - warning requires concrete blocker evidence.
   - blocked_not_verified is valid only with next proof step.
   - blocked_not_implemented is valid only for intentionally unavailable
     functionality and must not count as product proof.
   - planned is not product proof.

   Deliverables:

   - packages/proof/claim-firewall.mjs
   - scripts/claim-firewall.mjs
   - npm run claim:gate
   - MCP tool:
     - kernel.claims.verify

   Tests:

   - Rejects "done" without proof.
   - Allows "passed" with valid proof graph path.
   - Downgrades stale evidence.
   - Detects unsupported public release claims.
   - Detects unsupported client connection claims.

Layer 4: Looping And Self-Repair

7. Daily Operating Loop

   One command for practical daily use:

     sage operate --goal "..." --json

   Internal phases:

   1. Read repo state.
   2. Build task contract.
   3. Detect profile.
   4. Classify risk.
   5. Build proof graph seed.
   6. Plan implementation.
   7. Apply smallest change.
   8. Run impacted tests.
   9. Run risk gates.
   10. Run review.
   11. Run security/red-team if needed.
   12. Score.
   13. Repair if failed.
   14. Re-run until pass, budget, or blocker.
   15. Generate evidence-backed final report.

   Deliverables:

   - packages/operate/operate.mjs
   - bin/sage-operate-commands.mjs
   - sage operate
   - MCP tool:
     - kernel.operate.run

   Tests:

   - Can complete a small fixture bug fix end to end.
   - Stops on budget.
   - Stops on unsafe required approval.
   - Saves full proof graph.
   - Final report contains only evidence-backed claims.


8. Autonomous Repair Loop

   Repair loop must be bounded and evidence-driven.

   Repair policy:

   - Never hide failures.
   - Never retry more than configured budget.
   - Every repair attempt gets a proofId.
   - Every failed repair produces a postmortem candidate.
   - No destructive command without approval.

   Deliverables:

   - packages/operate/repair-loop.mjs
   - sage loop repair --from-proof <id>

   Tests:

   - Failing test repaired in fixture repo.
   - Bad repair reverted or superseded.
   - Flaky test detected and marked.
   - Repeated same failure creates postmortem.


9. Mutation Testing And Fault Injection

   Passing tests do not prove quality unless tests catch intentional faults.

   Deliverables:

   - packages/testing/mutation.mjs
   - npm run test:mutation
   - MCP tool:
     - kernel.testing.mutation

   Mutation targets:

   - condition flips
   - removed validation
   - skipped auth check
   - changed status code
   - missing error handling
   - deleted test assertion

   Tests:

   - Mutation run kills known bad mutations.
   - Surviving mutation lowers score.
   - Mutation evidence links to proof graph.
