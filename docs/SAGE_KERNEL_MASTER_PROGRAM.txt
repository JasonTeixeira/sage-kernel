SAGE KERNEL MASTER PROGRAM
MCP-FIRST OPEN SOURCE ENGINEERING OPERATING SYSTEM

Purpose
=======

This document is the master execution program for turning Sage Kernel into a
world-class open source MCP-native engineering operating system.

The product direction is:

  Sage Kernel is an MCP server first.
  The MCP server is the primary product interface.
  The local dashboard is an optional human cockpit.
  Hosted deployment is optional, not required.

The kernel should let a developer, team, or AI coding agent inspect projects,
run QA, audit code, manage approvals, execute jobs, generate production-ready
templates, and verify release readiness through a strict, documented, tested,
and auditable control plane.

North Star
==========

Build a kernel that is good enough for daily use and credible as a standalone
open source project.

The final project should satisfy these conditions:

1. A stranger can install it from a clean checkout.
2. A stranger can connect it to an MCP client.
3. A stranger can run the dashboard locally.
4. A stranger can run all tests, security scans, E2E checks, and stress tests.
5. Every important action is documented.
6. Every risky action has a permission and approval boundary.
7. Every route, MCP tool, job, and persistence path has automated coverage.
8. The repo has professional OSS docs, release process, and contribution flow.
9. The dashboard is useful as a daily operations cockpit.
10. The system can be extended through templates, tools, and eventually plugins.

PROGRAM 1: MCP PRODUCT CORE
===========================

Goal:
  Make the MCP server the primary product, not an afterthought.

Phase 1.1: Canonical MCP Entry Point
------------------------------------

Tasks:
  - Document that Sage Kernel is MCP-first.
  - Make `apps/mcp-server/src/server.mjs` the canonical runtime entry.
  - Add `npm run mcp:start` if missing.
  - Ensure the server can start from a clean checkout.
  - Ensure all tools load through the same runtime registry.

Code review checklist:
  - No duplicated tool-registration paths.
  - No hidden dependency on local maintainer paths.
  - No tool starts a long-running or mutating action without policy review.

Tests:
  - MCP server starts.
  - MCP smoke passes.
  - Tool manifest validates.

Verification:
  npm run mcp:validate
  npm run mcp:smoke
  npm run mcp:contracts

Exit criteria:
  A user can run the MCP server locally and call at least one safe read-only
  tool through a documented client path.

--------------------------------

      name
      description
      input schema
      output shape
      risk level
      permission scope
      examples
      failure modes
  - Add schema tests for malformed input.
  - Add schema tests for missing required input.
  - Add docs generation from the manifest.

Code review checklist:
  - No tool accepts arbitrary command strings.
  - No broad JSON blob where a strict object schema is possible.
  - No filesystem path input without root-boundary validation.

Tests:
  - Invalid input returns useful errors.
  - Valid input returns stable JSON.
  - Generated docs match the manifest.

Verification:
  npm run mcp:validate
  npm run mcp:contracts
  npm test

Exit criteria:
  The MCP tool surface is strict, documented, and contract-tested.

Code Review Standard
====================

Every PR or patch must answer:

  1. What behavior changed?
  2. What code paths are affected?
  3. What tests prove it?
  4. What security boundary is involved?
  5. What docs changed?
  6. What could fail in production?
  7. How do we roll back?

Review checklist:

  - No unrelated refactors.
  - No hardcoded local paths.
  - No secrets.
  - No process.exit inside reusable logic.
  - No broad catch blocks without useful fallback.
  - No shell execution from untrusted input.
  - No unbounded output to MCP clients.
  - No hidden network calls.
  - No undocumented env vars.
  - No route without test.
  - No MCP tool without schema and docs.
  - No job without retry/failure behavior.

Clean Code Standard
===================

Rules:

  - Keep modules small.
  - Keep functions testable.
  - Prefer explicit arguments over globals.
  - CLI files should be thin wrappers.
  - Business logic should be importable.
  - Use structured JSON for machine-readable output.
  - Validate inputs at boundaries.
  - Return clear errors.
  - Avoid clever code.
  - Avoid hidden behavior.
  - Avoid broad mutable state.

Refactor triggers:

  - A file is hard to test.
  - A script exits before logic can be imported.
  - A function mixes IO, policy, and formatting.
  - A module knows too much about unrelated layers.
  - Coverage gaps are caused by untestable structure.
