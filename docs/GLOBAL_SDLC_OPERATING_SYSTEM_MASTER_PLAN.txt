SAGE KERNEL GLOBAL SDLC OPERATING SYSTEM MASTER PLAN
====================================================

Purpose
-------

This document is the master execution plan for turning Sage Kernel into a
global, MCP-native, tool-agnostic, project-agnostic engineering operating
system.

The end state is not just a useful local tool. The end state is a portable
senior-engineering control plane that can be installed on any machine,
connected to compatible AI agents and MCP clients, pointed at any software
project, and used to inspect, plan, build, test, review, secure, release, and
improve production-grade systems through evidence-backed closed loops.

The operating principle is:

  Prove, do not claim.

Every meaningful action must produce evidence:

  - A detected project profile.
  - A written plan.
  - A machine-checkable definition of done.
  - Tests or validation output.
  - A review result.
  - A security result.
  - A release-readiness result.
  - A memory update.
  - An audit record.
  - A next-action list.

North Star
----------

Sage Kernel should become a global engineering and architecture operating
system for serious builders.

It should let a developer, team, or AI coding agent do the following on any
project:

  1. Understand the project.
  2. Detect the stack and quality requirements.
  3. Create a senior-level implementation plan.
  4. Build in small, reversible patches.
  5. Run the correct test suite for the project type.
  6. Review the diff like a senior engineer.
  7. Audit security and supply chain risk.
  8. Verify routes, APIs, jobs, persistence, and user workflows.
  9. Stress test relevant surfaces.
  10. Package and release safely.
  11. Record lessons, decisions, and recurring issues.
  12. Improve its own standards over time without losing control.

Quality Definition
------------------

The target phrase "100/100" does not mean perfect software. It means no known
critical gaps across the categories that matter for a real engineering system:

  - Installability
  - Portability
  - MCP compatibility
  - CLI usability
  - Dashboard usability
  - Architecture quality
  - Code quality
  - Test quality
  - Branch coverage
  - E2E coverage
  - Security
  - Supply-chain safety
  - Observability
  - Performance
  - Stress tolerance
  - Persistence correctness
  - Release readiness
  - Documentation
  - Contributor experience
  - Agent orchestration
  - Memory and learning
  - Self-audit and drift control
  - Long-term maintainability

Final Acceptance Standard
-------------------------

Do not call the system world-class unless all of these are true:

  1. Clean checkout install works.
  2. Global install works.
  3. MCP server starts and responds.
  4. At least one real external MCP client is manually connected and recorded.
  5. CLI doctor passes.
  6. Dashboard starts locally.
  7. All route and API checks pass.
  8. All MCP tool contracts pass.
  9. All runbooks validate.
  10. All project profiles validate against fixtures.
  11. All unit, integration, E2E, security, stress, and release gates pass.
  12. Real Postgres integration passes.
  13. Fresh-install verification passes in CI.
  14. Public package install is proven from npm.
  15. Signed release and provenance are proven.
  16. Long-duration soak testing is recorded.
  17. Security model is documented and tested.
  18. Agent orchestration has bounded retries, rollback, approvals, and audit.
  19. Memory has explicit write policy and drift control.
  20. Public docs, examples, screenshots, and demo assets exist.

Permanent Operating Loops
-------------------------

Every program in this plan must reinforce these loops:

  1. Intent Loop
     Clarify goal, scope, constraints, non-goals, and success criteria.

  2. Inspection Loop
     Detect stack, repo shape, routes, dependencies, risks, tests, docs, CI,
     security posture, and release posture.

  3. Planning Loop
     Produce a small, testable plan with rollback and proof requirements.

  4. Build Loop
     Implement changes in small patches with bounded scope.

  5. Verification Loop
     Run tests, coverage, type checks, linting, E2E, stress, security, and
     release checks.

  6. Review Loop
     Review architecture, code, tests, security, UX, performance, and release
     readiness using evidence.

  7. Hardening Loop
     Refactor, simplify, remove dead code, improve error handling, improve
     observability, and improve tests.

  8. Release Loop
     Fresh install, package, provenance, changelog, docs, CI, deployment proof,
     and rollback proof.

  9. Learning Loop
     Record decisions, repeated failures, standards, preferences, and lessons
     into memory and project instructions.

Program Map
-----------

There are 14 programs. Each program has 4 phases. Execute them in order unless
a phase is clearly independent and can be parallelized.

PROGRAM 1: BASELINE PROOF AND RELEASE STABILITY
===============================================

Goal:
  Lock the current project into a clean, repeatable, externally proven
  baseline before adding more intelligence.

Phase 1.1: Baseline Audit Snapshot
----------------------------------

Tasks:
  - Run the full local verification suite.
  - Record current coverage, CI status, stress results, and known gaps.
  - Confirm the worktree is clean before and after the audit.
  - Produce a release-proof snapshot document.

Deliverables:
  - Updated release proof notes.
  - Baseline scorecard.
  - Known-gaps list.

Verification:
  npm run test:coverage
  npm run coverage:critical -- /tmp/sage-coverage-output.txt
  npm run release:check
  npm run verify:fresh-install -- --worktree-copy
  npm run security:scan
  npm audit
  git diff --check

Exit criteria:
  All local gates pass and the baseline gap list is explicit.

Phase 1.2: CI Proof Lock
------------------------

Tasks:
  - Ensure CI runs quality gates, fresh install, and real Postgres integration.
  - Ensure CI captures the critical coverage ratchet.
  - Push changes and watch the remote run to completion.

Deliverables:
  - Passing GitHub Actions run URL.
  - CI status recorded in release proof.

Verification:
  gh run watch <run-id> --exit-status

Exit criteria:
  Remote CI passes on the current commit.

Phase 1.3: Public Release Readiness
-----------------------------------

Tasks:
  - Verify package metadata.
  - Verify public docs.
  - Verify release provenance workflow.
  - Verify npm package name and publishing method.
  - Document any blocked external steps.

Deliverables:
  - Release checklist.
  - Package install proof.
  - Publish-blocker list if npm auth or signing is missing.

Verification:
  npm run public:validate
  npm run release:provenance
  npm pack --dry-run
  npm whoami

Exit criteria:
  Release path is either proven or blocked by clearly named external
  credentials/configuration.

Phase 1.4: First Public Release Proof
-------------------------------------

Tasks:
  - Configure npm trusted publishing or NPM_TOKEN.
  - Configure signing if signed tags are required.
  - Create release tag.
  - Publish package with provenance.
  - Install from public npm in a clean temp project.

Deliverables:
  - Public package URL.
  - Signed tag or documented tag policy.
  - Clean npm install proof.

Verification:
  npm install -g <package-name>
  sage doctor --fast --json
  sage mcp smoke

Exit criteria:
  A stranger can install the package from npm and run the basic doctor and MCP
  smoke flow.

Non-Negotiable Engineering Rules
--------------------------------

  1. No unsupported world-class claims.
  2. No new workflow without tests.
  3. No mutating agent without approvals.
  4. No memory write without policy.
  5. No release without fresh install proof.
  6. No major feature without docs.
  7. No generated project without validation.
  8. No dashboard workflow without E2E proof.
  9. No security-sensitive action without audit record.
  10. No self-healing without retry limits and rollback.

Execution Method
----------------

For each phase:

  1. State the phase being executed.
  2. Inspect the current code.
  3. Implement the smallest complete slice.
  4. Add tests.
  5. Run focused tests.
  6. Run relevant gates.
  7. Update docs.
  8. Commit changes.
  9. Push if remote proof is needed.
  10. State what is done and what is left.

Final Iteration Rule
--------------------

After Program 14, do not stop. Run the full audit again.

If the score is below the target, create a new gap-closure program and repeat:

  audit -> prioritize -> implement -> test -> prove -> document -> release

The system is complete only when the remaining gaps are non-critical,
documented, accepted, and backed by evidence.
